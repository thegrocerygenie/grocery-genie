# Grocery Genie — Codebase Audit (2026-06-13)

## Verdict

**Functionally advanced, not launch-ready.** All 6 MVP API contract endpoints are
implemented and the scan→extract→structure→enrich→act pipeline is wired end-to-end
and proven by integration tests. A full auth system (Apple/Google/email, JWKS, token
denylist, soft-delete) landed since the last audit and closed the previous CRITICAL
"hardcoded mobile token" gap. **However, three deployment-blocking CRITICALs, several
launch-blocking HIGHs, and a completely empty LLM-accuracy harness (the product's stated
moat) prevent shipping.**

Build is green on both stacks — the blockers are correctness/deploy/security issues that
tests don't currently cover, not red builds.

## CRITICAL — deployment blockers (all verified directly)

### C1. Baseline Alembic migration cannot run on a fresh database
`backend/migrations/versions/20260430_0001_auth_softdelete_prefs.py`
`down_revision = None` makes this the first/only migration, but it consists solely of
`op.add_column("users", …)`, `op.add_column("receipts", …)`, `op.create_table("auth_events")`,
etc. There is **no `create_table` for `users`, `receipts`, `line_items`, `categories`,
`budgets`, `stores`, `notifications`, `user_item_mappings`**. `alembic upgrade head` on an
empty Postgres fails immediately ("relation users does not exist"). Only the test path works
because tests use `Base.metadata.create_all`. **Production / CI from scratch is broken.**
- Fix: generate a true initial migration (autogenerate against clean DB), set it
  `down_revision = None`, and make `20260430_0001` revise it. Add a CI job that runs
  `alembic upgrade head` against a fresh Postgres container + `alembic check`.

### C2. `lifespan` startup is empty — default categories never seeded in production
`backend/main.py` (lifespan body is `# Startup` then `yield`)
`seed_default_categories` is only called from test fixtures. On a fresh prod boot the
`categories` table is empty, so `category_lookup` in `receipt_service` resolves nothing and
**every receipt's line items come back uncategorized**; budget category summaries break too.
- Fix: in `lifespan` startup, open a session and `await seed_default_categories(session)`.

### C3. Hardcoded JWT signing secret with no production guard
`backend/app/core/config.py:50` — `jwt_secret = "dev-secret-change-me-32+bytes-of-random-data"`
No startup assertion anywhere forces this to be overridden (verified: only uses are
encode/decode in `security.py`). If `GG_JWT_SECRET` is unset in any environment, all access
tokens are signed with a well-known default → **anyone can forge a token for any `user_id`.**
- Fix: fail fast at startup when `not debug and jwt_secret == <default>`. Apply the same
  guard to `resend_api_key`, `apple_private_key`, `llm_api_key`.

## HIGH — launch blockers

### H1. LLM accuracy harness is empty — core IP has zero automated coverage
`backend/tests/llm_validation/receipts/` (only `.gitkeep`); `ground_truth/` has 1 of the
required 30+ receipts. All accuracy/consistency/regression tests `pytest.skip`. Five DoD
gates (≥90% field accuracy, 0% hallucination, ≥95% recall, ≥95% consistency, RC-05
multi-language) are **unverifiable**. CLAUDE.md names this the competitive moat and a
mandatory pre-merge gate for prompt changes.
- Fix: add ≥10 receipt image + ground-truth pairs (US/UK DD-MM/DE comma-decimal/FR/SR);
  run accuracy tests in CI with a masked `GG_LLM_API_KEY`; add an adversarial-prompt fixture.

### H2. Backend ignores the per-user notification preferences the mobile app now writes
`backend/app/services/budget_service.py` `check_thresholds` — iterates the global
`budget_alert_thresholds` and never loads the `User` row, so `notif_threshold_50/80/100`
(persisted via `PUT /api/users/me`, toggled in the mobile settings screen) are write-only
dead columns. A user who disables the 80% alert still gets it.
- Fix: load the user, filter thresholds by their enabled flags.

### H3. Category correction flow sends slug IDs the backend rejects (422)
`mobile/src/components/CategoryPickerSheet.tsx` emits `DEFAULT_CATEGORIES` slugs
(`'groceries'`) as `category_id`; `review.tsx` puts that in the PATCH body where the backend
expects a UUID → 422. Also `getCategoryMeta` (dashboard/history) keys `CATEGORY_BY_ID` by
slug while backend returns UUIDs, so **every category renders as "Other".** The new
`CategoryPickerSheet` (commit 429e0a4) reproduced the old defect.
- Fix: thread `useCategories()` UUIDs into the sheet and `getCategoryMeta`; keep slugs only
  as an offline fallback.

### H4. Receipt images are never served — broken thumbnails for every confirmed receipt
`backend/main.py` has no `StaticFiles` mount for `/uploads`, but `Receipt.image_url` points at
`/uploads/receipts/{uuid}.jpg`. History re-opens 404 the image.
- Fix: mount `/uploads` (gated to debug) or serve via an authenticated `GET
  /api/receipts/{id}/image` with an ownership check (preferred — avoids the LOW IDOR below).

### H5. Event-driven architecture is inert; `confirm_receipt` is synchronous
`backend/app/events/dispatcher.py` has **zero registered handlers** — every `dispatch()`
walks an empty list. `confirm_receipt` fires `ReceiptConfirmed` (no-op) *and* directly
instantiates `BudgetService`/`NotificationService` — the exact coupling the event model
exists to avoid. Misleads contributors who assume handlers fire.
- Fix: wire a `ReceiptConfirmed` handler in a `wiring.py` imported at startup, or delete the
  dispatcher and document the decision.

### H6. LLM robustness gaps (CLAUDE.md mandates unmet)
- `LiteLLMCategoryAssigner.assign` returns raw `json.loads` with **no Pydantic validation**
  (`provider.py`), unlike the receipt extractor — violates "validate all LLM output before
  persistence."
- The three schema validators (`check_price_tolerance` / `check_date_not_future` /
  `check_subtotal_tolerance`, `llm/schemas.py`) detect drift then `pass` — **dead validation
  layer**; future-dated/locale-skewed receipts ship to the DB.
- `receipt_service` swallows category-assignment errors with `except Exception: pass` (no log,
  no metric); **no rule-based fallback exists** despite CLAUDE.md requiring one.
- `date.fromisoformat(extraction.date)` has no fallback → 500 on non-ISO LLM output.
- Fix: validate category output with a Pydantic model; make validators downgrade confidence
  or attach warnings; log + emit on assignment failure; add a keyword `RuleBasedCategoryAssigner`;
  guard date parsing.

### H7. New auth security gaps (verified by security reviewer)
- Password reset does **not** invalidate active refresh tokens (`auth.py`) — stolen refresh
  token survives a reset for up to 30 days.
- Apple token-exchange `id_token` decoded with `verify_signature=False` (`apple_auth.py`).
- Raw verify/reset/email-change tokens **logged in plaintext** (`email.py`) — log capture =
  account takeover.
- Sign-in lockout counter is cumulative; configured `signin_lockout_window_seconds` is
  **unused dead config** (`auth.py` / `config.py`).
- Unverified-email accounts get full API access immediately (`security.py` `get_current_user`
  never checks `email_verified_at`).

## MEDIUM (selected)

- **Fake metric shown to users**: `mobile/app/trend.tsx` renders a hardcoded `priorAvg*1.15`
  delta that always displays "−13% vs prior" regardless of real data. Pull before launch.
- **CORS** `allow_origins=["*"]` + `allow_credentials=True` (`main.py`) — invalid combo,
  unscoped policy. Drive from `settings.cors_origins`, fail closed in prod.
- **No upload size cap** on `/api/receipts/scan` (`receipts.py` `await file.read()`) — OOM DoS.
  Reject >10 MB with 413.
- **Rate limiter** in-memory, keyed by raw JWT string → unbounded dict growth as tokens rotate;
  reads `request.client.host` ignoring proxy headers → ineffective behind a load balancer.
- **`os.environ["ANTHROPIC_API_KEY"]` mutation** per-construction in `provider.py` (redundant —
  per-call `api_key` already passed; mutates shared process state).
- **String-matching for HTTP status** (`"not found" in str(e)`, `"past 30-day" in str(e)`) across
  routes — reworded message silently breaks status codes. Use typed exceptions.
- **Duplicate detection** uses a 24h `created_at` window (misses 25h-late re-scans, false-flags
  two same-day same-total trips). **Retroactive category update** lacks a `Receipt.status` filter.
- **N+1 on dashboard**: `get_spending_trend` / `get_period_dates` issue ~12 queries per load.
- **`temperature=0` not enforced** (env-overridable). **Runtime `assert`** for extractor
  invariant (compiled out under `-O`). **`process_receipt_async` Celery task is an empty stub.**
- **Mobile**: `ReceiptScanResponse` type missing `duplicate_warning`; `scanReceipt` fetch has no
  timeout/abort; `authStore.safeSet` silently swallows Keychain write failures;
  `app.json` `userInterfaceStyle:"automatic"` with no dark palette → broken dark mode;
  `useSyncQueue` lacks a concurrency guard (double-submit on rapid reconnect); strings not
  externalized for i18n.

## LOW (selected)

- IDOR on `/uploads` once mounted (no ownership check) — resolve via H4's authenticated route.
- `database_echo=true` in prod would log auth-token SQL params — refuse echo when `!debug`.
- Apple Sign In (mobile) generates/verifies no nonce.
- `top-items.tsx` index-derived keys; `trend.tsx` legend truncated to 4/8 categories silently.
- Local untracked `backend/app/.env.local` holds out-of-scope Stripe keys — delete it.
